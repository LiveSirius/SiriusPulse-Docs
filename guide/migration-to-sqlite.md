# 迁移到 SQLite 数据库

本文档说明如何将现有数据迁移到 SQLite 数据库，以便利用 SQLite 的高效存储和查询能力。迁移脚本 `scripts/migrate_to_sqlite.py` 提供了一个自动化流程，用于将原有文件或内存中的记录安全转移到 SQLite 数据库中。

## 背景

在早期版本中，数据以 JSON 文件或内存数据结构存储。随着系统扩展，需要更稳定、可查询的持久化方式。SQLite 提供了轻量级数据库解决方案，支持事务、索引和增量查询，适合单机部署场景。

## 前提条件

- 已安装 Python 3.8+ 环境。
- 已正确配置 `sirius_pulse` 项目依赖（参见 `requirements.txt`）。
- 原始数据目录存在且格式正确（通常位于 `~/.sirius_pulse/data/` 或自定义路径）。
- 如需回滚，请提前备份原始数据。

## 使用方式

迁移脚本通过命令行调用，基本语法如下：

```bash
python scripts/migrate_to_sqlite.py [选项]
```

### 常用选项

| 选项 | 说明 |
|------|------|
| `--source PATH` | 指定原始数据的根目录。默认为自动查找的默认数据目录。 |
| `--db PATH` | 指定输出的 SQLite 数据库文件路径。默认为 `~/.sirius_pulse/data/memory.db`。 |
| `--dry-run` | 预览模式，仅显示将要执行的操作，不实际写入。 |
| `--force` | 强制覆盖已存在的数据库文件（忽略已有数据警告）。 |
| `--verbose` | 输出详细日志。 |

### 示例

```bash
# 预览迁移操作
python scripts/migrate_to_sqlite.py --dry-run

# 执行默认迁移
python scripts/migrate_to_sqlite.py

# 指定源路径和数据库路径
python scripts/migrate_to_sqlite.py --source /data/legacy --db /data/new/db.sqlite
```

## 迁移流程

1. **数据读取**：脚本遍历源目录，加载所有符合条件的记录（如 persona 配置、记忆、传记等）。
2. **格式转换**：将原有 JSON 或自定义数据结构映射为 SQLite 表结构。
3. **写入数据库**：使用事务批量插入，确保原子性。
4. **验证**：对写入的数据进行一致性检查，输出迁移统计。

### 迁移内容

| 数据类别 | 源文件/内存位置 | 目标表 |
|----------|----------------|--------|
| 人格配置 | `persona.json` | `personas` |
| 别名索引 | `index.json` | `aliases` |
| 传记卡片 | `qq_*.json` | `biography_cards` |
| token 使用记录 | `token_usage.json` 或内存 | `token_usage` |
| 日志数据 | 内存 / 文件 | `logs` |

> 注：具体迁移的数据类别取决于当前系统的存储实现。脚本会自动检测并迁移所有支持的数据类型。

## 回滚

如果迁移后出现数据异常，可以恢复至旧文件模式：

1. 确保原始数据未被删除（脚本默认不会删除源文件）。
2. 删除或重命名生成的 SQLite 数据库文件。
3. 重启系统，系统将自动回退到文件存储模式（如果代码支持）。

详细回滚步骤请参考 `回滚指南` 文档（如有）。

## 错误处理

- 如果脚本在迁移过程中失败（如磁盘空间不足、数据损坏），事务会回滚，数据库将保持不变。
- 日志中包含详细的错误信息，请根据提示解决后重试。
- 如遇格式不兼容的数据，脚本会跳过并记录警告。

## 注意事项

1. **备份**：迁移前请务必备份原始数据。虽然迁移脚本设计为不破坏源数据，但意外情况仍有可能发生。
2. **数据库版本**：确保 SQLite 版本 >= 3.35（推荐 3.40+）以支持所有功能。
3. **性能**：大量数据迁移可能需要较长时间，建议先使用 `--dry-run` 预览。
4. **环境**：不要在系统正在运行时执行迁移，请先停止相关服务进程。

## 常见问题

**Q: 迁移后系统无法启动？**
A: 检查数据库文件是否存在且权限正确，如 `~/.sirius_pulse/data/memory.db`。若问题持续，请回滚并检查日志。

**Q: 如何验证迁移成功？**
A: 运行 `sqlite3 <db_path> .tables` 查看表是否存在，并使用 `SELECT COUNT(*) FROM <table>` 检查记录数。

**Q: 可以多次运行迁移吗？**
A: 可以，但第二次运行会先检测目标数据库是否存在并跳过已迁移的表（除非使用 `--force`）。
